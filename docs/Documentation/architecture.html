Architecture

A 14-crate Rust workspace built with Ratatui, Tokio, and a trait-based provider abstraction.

Workspace Overview

OpenAnalyst CLI is structured as a Rust workspace with 14 crates, each with a focused responsibility:

rust/crates/
├── openanalyst-cli     # Binary entry point, arg parsing, mode selection
├── api                 # Multi-provider API client (7 providers)
├── runtime             # Conversation engine, sessions, config, OAuth, MCP
├── tui                 # Full-screen terminal application
├── tui-widgets         # Custom widgets: markdown, tool cards, input, spinner
├── commands            # 51+ slash command definitions and parsing
├── tools               # 19 built-in tool implementations
├── orchestrator        # Multi-agent lifecycle, model routing, MOE
├── events              # Shared event types for TUI ↔ backend channels
├── plugins             # Plugin system (install, enable, disable, hooks)
├── server              # HTTP/SSE server (Axum-based)
├── lsp                 # Language Server Protocol integration
├── openanalyst-agent   # Headless autonomous agent runner
└── compat-harness      # TypeScript manifest extraction for parity

Data Flow

The system follows a layered architecture with clear separation of concerns:

1

Input Layer (TUI / REPL / CLI)

User input is captured by the TUI event loop (crossterm) or REPL readline. Slash commands are parsed by the commands crate. Regular prompts are forwarded to the runtime.

2

Runtime Layer (ConversationRuntime)

The core agentic loop. Assembles the system prompt, manages session state, sends messages to the API client, processes responses, and dispatches tool calls. The runtime is generic over C: ApiClient and T: ToolExecutor traits.

3

API Layer (ProviderClient)

Translates the provider-agnostic conversation format into provider-specific API calls. Handles SSE streaming, token counting, and response parsing for all 7 providers.

4

Tool Layer (ToolExecutor)

Executes tool calls from the agent. Each tool validates its parameters, checks permissions, performs the operation, and returns structured results.

5

Orchestrator Layer

Manages multi-agent workflows. Spawns sub-agents with their own runtime instances, routes models per task type, and bridges async TUI events with sync runtime operations via Tokio channels.

Startup Flow

Terminal Setup

Enter raw mode, alternate screen, install panic handler for graceful restoration.

Config & Auth

Merge config from all sources. Load saved OAuth tokens. Resolve provider and model.

System Prompt

Assemble system prompt from OPENANALYST.md, project context, and model-specific guidance.

Plugins & MCP

Initialize enabled plugins. Connect to configured MCP servers. Register discovered tools.

TUI Render

Draw startup banner, status bar, empty chat. Begin event loop.

Event Loop

Listen for user input, orchestrator events, and stream updates. Process until exit.

Building

# Debug build
cd rust && cargo build

# Release build (optimized)
cd rust && cargo build --release

# Run directly
cargo run --release -p openanalyst-cli

# Run tests
cargo test --workspace
